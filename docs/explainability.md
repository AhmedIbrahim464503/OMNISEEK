# Explainable Search Results

This document describes the explainability layer, rationale generation logic, and metadata structures that explain why results matched.

## Purpose

Black-box search engines make it difficult for users to understand why a document matched a query, especially when semantic vectors pull conceptually related terms. The explainability layer translates internal similarity metrics and query patterns into clear, human-readable explanations to build trust and improve the user experience.

## Design

### 1. Explainability Metadata
For every search result, the response includes the following explanation block:
```json
{
  "reason": "Strong match containing both matching keywords (score: 0.73) and contextual conceptual themes (score: 0.91) inside 'report.pdf'.",
  "semantic_score": 0.91,
  "keyword_score": 0.73,
  "vector_score": 0.91
}
```

### 2. Explanation Generation Logic
The logic in `ExplainabilityService` dynamically drafts match reasons based on the active search profile:
- **Fast Mode**: Generates explanations highlighting the high semantic/conceptual similarity (e.g. `Highly relevant semantic match (0.85 similarity)...`).
- **Balanced Mode**: 
  - If a document matches both keyword and vector pathways, it explains the combined lexical-semantic match.
  - If it matches only one pathway, it explains that it is a conceptual/synonym match or a keyword-based match.
- **Accurate Mode**: Highlights that the match has been verified and refined by the Cross-Encoder model.

## Tradeoffs

- **Static Template-Based Rationales**: Explanations are template-driven for speed and safety. While this guarantees execution times under 1ms, it is less natural than explanations generated by large LLMs.
- **No LLM Integration**: Generating matching explanations using local LLMs (like Llama) would introduce seconds of latency. Template-based explanation generation maintains search speeds under 500ms.

## Future Improvements

- **Highlight Query Words**: Bold matching keyword terms directly in the content string returned to the client.
- **LLM-in-the-loop (Async)**: Add an optional asynchronous workflow that uses a local LLM to summarize matching rationales for selected documents.
