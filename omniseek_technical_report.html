1. Project Vision & Problem Analysis

OmniSeek is an enterprise-ready, multi-modal semantic search platform designed to ingest, process, understand, index, and retrieve information across text documents, audio recordings, and video files. Traditional keyword-based search algorithms fail to scale when data modalities expand beyond plain text, resulting in fragmented information silos. OmniSeek addresses this by constructing a unified high-dimensional vector space where queries and multi-modal content chunks are compared semantically.

1.1 Problem Statement

Traditional search indexers suffer from systemic limitations across different media modalities:

I. Text Search Semantic Gap

Lexical systems rely strictly on token matching. They cannot resolve conceptual synonyms or grammatical variations. For instance, a query for "car" fails to return documents containing "automobile", "vehicle", or "sedan" despite sharing the same semantic concept. Furthermore, keyword match indexers lack context sensitivity, rendering them incapable of resolving homographs or complex query phrases.

II. Audio Opacity & Temporal Search Blindness

Audio binary streams are structurally opaque. Search engines cannot index acoustic contents directly. To find a specific topic, such as a "machine learning discussion", within a file like podcast.mp3, the system must transcribe the spoken speech, generate localized embeddings, and map transcripts to temporal indices. Manual scanning is inefficient and fails to scale.

III. Video Multi-Stream Synchronization Complexity

Video files coordinate visual frames, speech transcripts, and time. Standard systems cannot process these streams together. Consequently, queries like "Find the moment where a person drinks milk" cannot be resolved without visually extracting and embedding frames alongside transcript indexing, mapping the query to a unified coordinate space, and resolving the exact timestamps.

1.2 Project Objectives

  • Multi-Modal Ingestion Framework: Process text, audio, and video uploads through distinct processing pipelines.
  • Unified Semantic Vector Space: Map text, transcribed speech, and visual frames into a synchronized high-dimensional space.
  • Temporal-Specific Retrieval: Pinpoint exact timestamps in audio and video files where queries match.
  • Cross-Modal Query Operations: Match text inputs against visual features in video frames, audio transcriptions, and written documents.
  • Scalable Architecture: Implement async processing, HNSW graph vector indices, and caching for real-world deployment.

2.2 System Component Descriptions

  • FastAPI Backend: Acts as the system orchestrator. It manages database transactions, executes quick search lookups, coordinates Celery task dispatches, and handles rate limiting.
  • Celery Ingestion Worker: An isolated worker process that loads deep learning models (Faster-Whisper, CLIP, and BGE) to run media file pipelines.
  • Redis Message Broker: Acts as the broker for Celery workloads. It also serves as the application's caching layer, caching search results and database metrics.

  • Next.js 14 Frontend Client: Provides the user dashboard, search results viewer, and timestamp-based media playbacks.

3.2 Vector Indexing Configuration

To enable sub-linear nearest-neighbor lookup speeds over large vector sets, pgvector uses the **Hierarchical Navigable Small World (HNSW)** index. HNSW constructs a multi-layered graph where the bottom layer contains all elements and higher layers form highway networks.

Standard cosine distance queries calculate similarities using the equation:

DistanceCosine(u, v) = 1.0 - (u · v) / (||u||2 · ||v||2)

Because all vector embeddings in OmniSeek are normalized to a unit length of 1.0 (||u||2 = 1.0), this simplifies to:

DistanceCosine(u, v) = 1.0 - (u · v)

This reduction allows the system to compute similarities using dot-product operations. The HNSW index is defined as follows: 

CREATE INDEX idx_asset_chunks_embedding 
ON asset_chunks 
USING hnsw (embedding vector_cosine_ops);

5. AI Intelligence & Embedding Alignment

OmniSeek uses deep learning models to convert multi-modal documents, audio transcriptions, and image frames into unified vector spaces.

5.1 Thread-Safe Model Singleton Pattern

Loading neural networks for each request causes high memory usage and latency. To prevent this, OmniSeek loads models once using a thread-safe singleton design pattern. When the application starts, locks prevent multiple threads from loading duplicates. The single model instance is then shared across worker threads.

5.2 High-Dimensional Alignment via Text Vector Zero-Padding

OmniSeek stores vector embeddings in a pgvector column with a fixed size of 512 dimensions. However, the text embedding model (such as all-MiniLM-L6-v2) outputs 384 dimensions, while the visual CLIP model (ViT-B/32) outputs 512 dimensions.

To make text and visual vectors compatible, the system pads the 384-dimensional text vectors with 128 zeros and applies L2 normalization: 

import numpy as np

def align_and_normalize_vector(raw_vector: np.ndarray, target_dim: int = 512) -> np.ndarray:
    raw_dim = raw_vector.shape[0]
    if raw_dim == target_dim:
        # Vector is already aligned, normalize to unit length
        return raw_vector / np.linalg.norm(raw_vector)
    
    # Zero-pad the remaining dimensions
    padding_size = target_dim - raw_dim
    padded_vector = np.pad(raw_vector, (0, padding_size), 'constant')
    
    # Re-normalize to ensure Unit Length
    norm = np.linalg.norm(padded_vector)
    if norm > 0:
        return padded_vector / norm
    return padded_vector
Mathematical Proof of Cosine Distance Preservation

Let u, v in Rd represent two 384-dimensional vectors. Their L2-normalized versions are û = u / ||u||2 and v̂ = v / ||v||2. Their cosine similarity is defined as:

SimCosine(û, v̂) = û · v̂ = Σ(i=1 to d) ûi v̂i

Let ũ, ṽ in RD (where D = 512) represent the zero-padded vectors:

ũ = [û1, ..., ûd, 0, ..., 0],      ṽ = [v̂1, ..., v̂d, 0, ..., 0]

The L2-norm of the padded vector ũ is:

||ũ||2 = √(Σ(i=1 to d) ûi2 + Σ(j=d+1 to D) 02) = ||û||2 = 1.0

Because ||ũ||2 = 1.0 and ||ṽ||2 = 1.0, the cosine similarity of the padded vectors is:

SimCosine(ũ, ṽ) = ũ · ṽ = Σ(i=1 to d) ûi v̂i + Σ(j=d+1 to D) (0 × 0) = û · v̂

This shows that zero-padding preserves the original cosine similarity and distance properties exactly without distortions.

6. Semantic Retrieval & Hybrid Search Engine

OmniSeek uses a multi-stage search engine to combine semantic vector matching, lexical keyword queries, and neural reranking.

6.1 Weighted Hybrid Search Execution

While vector search captures semantic concepts, it can overlook exact keyword matches like product codes or names. Hybrid search resolves this by combining pgvector cosine similarity search with PostgreSQL Full-Text Search (FTS).

The lexical score is normalized using a soft saturation curve to prevent outlier keyword hits from skewing results:

ScoreFTS = ts_rank(content, query) / (ts_rank(content, query) + 1.0)

The vector similarity score is calculated from the cosine distance:

ScoreSemantic = 1.0 - DistanceCosine

These scores are combined using a weighted average:

ScoreHybrid = (w × ScoreSemantic) + ((1.0 - w) × ScoreFTS)

Where the weight coefficient w is set to 0.7, prioritizing semantic context while maintaining lexical matches.

6.2 Neural Cross-Encoder Reranking

In **Accurate Search Mode**, the search engine uses a Cross-Encoder model (cross-encoder/ms-marco-MiniLM-L-6-v2) to rerank the top 50 hybrid candidates.

Unlike bi-encoders, which generate separate embeddings for queries and targets, the Cross-Encoder processes the query and document chunk together. This captures deeper attention patterns. The raw output logit from the network is converted into a score between 0 and 1 using a sigmoid function:

ScoreRerank = σ(logit) = 1 / (1 + e-logit)

Candidates with scores below 0.30 are filtered out to improve result quality.

7. Evaluation Framework & Performance Benchmarks

OmniSeek includes an automated evaluation pipeline to measure search quality. It runs test queries and compares system results against defined ground-truth assets.

7.1 Information Retrieval Metrics

  • Precision@K: Measures accuracy by calculating the proportion of relevant chunks in the top K returned results:
    Precision@K = (1 / K) × Σ(i=1 to K) Ri
    Where Ri in {0, 1} indicates the relevance of the result at rank i.
  • Recall@K: Measures completeness by calculating the fraction of total ground-truth chunks returned in the top K results:
    Recall@K = (Σ(i=1 to K) Ri) / |G|
    Where |G| is the total number of relevant ground-truth chunks.
  • Mean Reciprocal Rank (MRR): Evaluates position quality by scoring based on the rank of the first correct match:
    MRR = 1 / min { i | Ri = 1 }
    If no correct match is found within the results, the MRR score is 0.
  • NDCG@K: Normalized Discounted Cumulative Gain measures query sorting relevance against ideal ordering limits, discounting lower-ranked results:
    DCG@K = Σ(i=1 to K) Ri / log2(i + 1)
    IDCG@K = Σ(i=1 to min(K, |G|)) 1 / log2(i + 1)
    NDCG@K = DCG@K / IDCG@K

8. Production Hardening & Operational Resiliency

OmniSeek includes security, rate-limiting, and caching configurations designed for production deployment.

8.1 Security & Authentication Policies

  • JWT Access Controls: API endpoints are secured using JSON Web Tokens (JWT) with HMAC-SHA256 signatures. The system supports two user roles: ADMIN (access to uploads and evaluations) and USER (read-only search access).
  • CORS Policy Boundaries: Cross-Origin Resource Sharing (CORS) limits requests to trusted clients, blocking unauthorized domains.

8.2 Redis Caching & Rate Limiting

Rate Limiting Mechanism

The API gateway uses a Redis-based token bucket algorithm to rate-limit requests. Limits are configured by user tier and IP address, protecting the system from brute-force attempts:
- Upload endpoints: 10 attempts per minute.
- Search queries: 100 queries per minute.

Search Results Caching: Redis caches query results using the query string, search mode, and filters as the cache key. When data is updated or assets are deleted, the cache is invalidated to keep search results fresh.

8.3 System Observability & Monitoring

OmniSeek exposes two metrics endpoints to monitor system health:

  • GET /health: Fast status check that validates the database and Redis connections. Returns: 
    {"status": "ok", "service": "omniseek-backend"}
  • GET /metrics: Prometheus-compatible metrics endpoint tracking API latencies, search database response times, model inference times, and task durations.

8.4 Disaster Recovery & Backups

To prevent data loss, OmniSeek includes a backup process that runs daily: 

# Backup database structure and vector data
pg_dump -h db -U postgres -d omniseek -F c -b -v -f /backups/omniseek_db_daily.dump

# Sync storage directories to an offsite S3-compatible bucket
